## Architecture: single image, three processes via supervisord

Upstream bilibili-rag ships as two separate processes with no Dockerfile:

- **FastAPI backend** on `:8000` (Python; bilibili API + ASR + RAG)
- **Next.js frontend** on `:3000` (Next.js 16; chat UI, calls backend over HTTP)

The patch adds a multi-stage Dockerfile and packs both into one image
fronted by nginx:

```
                 в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ container в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
   browser в”Җв”Җ>   nginx :80   в”Җв”Җ/api/*в”Җв”Җ>   uvicorn :8000  в”Ӯ
                              в”Җв”Җ   /  в”Җв”Җ>   node :3000     в”Ӯ
                 в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

- `nginx` strips the `/api` prefix and proxies to FastAPI on `127.0.0.1:8000`.
- Everything else goes to the Next.js standalone server on `127.0.0.1:3000`.
- Frontend bundle is built with `NEXT_PUBLIC_API_URL=/api`, so the browser
  never speaks to backend ports directly вҖ” only nginx is exposed.
- supervisord runs all three (uvicorn, node, nginx) with `nodaemon=true` as
  PID 1 via `lazycat-entrypoint.sh`.

`patches/01-lazycat-dockerfile.patch` adds:

- `Dockerfile` вҖ” three-stage build (frontend builder, backend deps builder,
  final runtime). Final image installs `nginx + supervisor + ffmpeg + nodejs`.
- `.dockerignore` вҖ” strip vendor noise from the build context.
- `nginx.conf` вҖ” `/api/` strip-prefix proxy + Next.js root proxy + `/health`.
- `supervisord.conf` вҖ” backend, frontend, nginx programs with stdout/stderr
  pinned to the container log stream.
- `lazycat-entrypoint.sh` вҖ” ensures `/app/data` and `/app/logs` exist (they
  are bind targets), then `exec`s supervisord.
- `frontend/next.config.ts` вҖ” adds `output: "standalone"` so the runtime
  image only carries `server.js` + pruned `node_modules`, not the full
  `.next/` tree.

## Deploy params drive runtime config

Upstream config is read via `pydantic_settings` from env vars. The
manifest injects:

- `DASHSCOPE_API_KEY` (required, install-time)
- `LLM_MODEL`, `EMBEDDING_MODEL`, `ASR_MODEL` (optional, sensible defaults)

plus fixed values for `OPENAI_BASE_URL`, `DASHSCOPE_BASE_URL`,
`DATABASE_URL`, `CHROMA_PERSIST_DIRECTORY`, `APP_HOST`, `APP_PORT`, `DEBUG`.

There is no in-app settings panel (unlike Pixelle-Video), so all config is
collected at install time. Re-installing/upgrading lets the user change the
values; the persisted SQLite/Chroma data is preserved across upgrades via
the `/lzcapp/var/persist` bind.

## Persistent volumes

- `/lzcapp/var/persist:/app/data` вҖ” SQLite (`bilibili_rag.db`) +
  ChromaDB collection (`chroma_db/`)
- `/lzcapp/var/logs:/app/logs`   вҖ” loguru-rotated app logs (10 MB / 7 days)

## Icon

`lazycat/icon.png` is cropped from upstream's
`vendor/bilibili-rag/assets/screenshots/home.png` (BiliMind brand-mark,
dark rounded square with white burger). Re-derive from upstream вҖ” do
NOT magick-paint a synthetic placeholder:

```sh
magick vendor/bilibili-rag/assets/screenshots/home.png \
  -crop 125x125+40+260 +repage \
  -filter Lanczos -resize 360x360 /tmp/mark_big.png
magick -size 512x512 xc:'#F4F0E8' /tmp/mark_big.png \
  -gravity center -compose over -composite lazycat/icon.png
```

If upstream redesigns the brand-mark, re-take a fresh `home.png` first;
adjust the `-crop` offset/size to bound the new mark.

## Patches workflow

```sh
# Apply patches into vendor for live editing:
git apply patches/01-lazycat-dockerfile.patch -p1 --directory=vendor/bilibili-rag

# Make changes...
vim vendor/bilibili-rag/Dockerfile

# Stage new files so git diff sees them:
git add -N vendor/bilibili-rag/<any-new-files>

# Regenerate the patch:
git diff --no-color --relative=vendor/bilibili-rag \
  vendor/bilibili-rag/ > patches/01-lazycat-dockerfile.patch

# Restore vendor for clean commit (only patches/* should be tracked):
git checkout HEAD -- vendor/bilibili-rag/frontend/next.config.ts
rm vendor/bilibili-rag/Dockerfile vendor/bilibili-rag/.dockerignore \
   vendor/bilibili-rag/nginx.conf vendor/bilibili-rag/supervisord.conf \
   vendor/bilibili-rag/lazycat-entrypoint.sh

# Verify:
git apply --check patches/01-lazycat-dockerfile.patch \
  -p1 --directory=vendor/bilibili-rag
```

CI applies patches in lexical order (`patches/*.patch`) inside lazycat-ci's
docker job before `docker build`, with `--directory=vendor/bilibili-rag`.

## Bootstrap (first-time appstore submission)

`bootstrap-app.yml` is `workflow_dispatch` with `version` (required) +
`app_id` (optional, skip `/app/create` on retry) inputs. Calls
`lazycat-ci/lpk-build.yml` then `lazycat-ci/bootstrap.yml`.

- `/app/create` returns 500 on duplicate package вҖ” pass `app_id` workflow
  input to skip create on retry.
- After deleting an app from the lazycat developer center to redo
  bootstrap, bump `version` (e.g. `0.1.0` вҶ’ `0.1.1`) before re-running;
  the appstore enforces strict monotonic semver per package id.

The bootstrap run does **not** publish a GitHub Release; grab its lpk
from the workflow artifact instead:

```sh
gh run download <RUN_ID> -R microlazy-apps/lazy-bilibili-rag \
  -n lpk -D /tmp/lpk
```

## Updating bilibili-rag upstream

```sh
git subtree pull --prefix=vendor/bilibili-rag \
  https://github.com/via007/bilibili-rag.git main --squash

# Re-verify the patch still applies:
git apply --check patches/01-lazycat-dockerfile.patch \
  -p1 --directory=vendor/bilibili-rag
```

If upstream changes the FastAPI entry path (`app.main:app`), the
`requirements.txt` location, or the frontend dir layout, edit the patch
(re-derive via the patches workflow above) before tagging a new release.

## Local lpk build (smoke test)

Requires `lzc-cli`, `envsubst` (gettext), and docker on PATH.

```sh
# one-time
git clone https://github.com/microlazy-apps/lazycat-ci ~/lazycat-ci

# in this repo:
bash ~/lazycat-ci/scripts/build-lpk.sh \
  --image lazy-bilibili-rag:dev \
  --version 0.0.0-dev \
  --docker-context ./vendor/bilibili-rag \
  --patches-target vendor/bilibili-rag
# вҶ’ lazycat/cloud.lazycat.app.bilibili-rag-0.0.0-dev.lpk
```

`lzc-cli` will emit `[lint] App Store submission requires
services.main.image to start with registry.lazycat.cloud` вҖ” benign for
local smoke tests; the lpk is still produced and installable via
`lpk-manager install`.

The first build is heavy: it fetches Node 20 + nginx + ffmpeg + the
Python venv (chromadb / langchain / openai / dashscope etc.) and compiles
the Next.js frontend. Cache hits on subsequent builds are good as long as
`requirements.txt` and `frontend/package.json` don't change.
